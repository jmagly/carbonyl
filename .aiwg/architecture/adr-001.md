# ADR-001: Daemon Architecture over In-Process Browser

**Status**: Accepted
**Date**: 2026-04-02
**Deciders**: roctinam

## Context

The automation layer needs to maintain long-lived browser sessions that survive client script restarts -- for example, staying logged into a site across multiple agent invocations without re-authenticating. A naive approach of spawning a new Carbonyl process per script invocation loses all in-memory browser state (DOM, WebSocket connections, pending XHRs) on every exit, forcing full page reloads and re-authentication. The architecture must also support LLM agents that operate in short, iterative turns rather than single long-running scripts.

## Decision

Use a persistent Unix-domain-socket daemon (`os.fork()` + `os.setsid()`) that owns the Carbonyl process and exposes its full API over a newline-delimited JSON wire protocol. Clients connect, issue commands, and disconnect without affecting the browser process. Mode selection (direct vs. daemon) is automatic: `open()` checks for a live daemon socket and transparently delegates if one exists.

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|----------------|
| **Subprocess per call** | Loses all browser state between invocations; every call pays full startup + page-load cost; cannot maintain authenticated sessions across agent turns. |
| **Embedded / in-process mode only** | Ties browser lifetime to the calling script; a script crash kills the browser and loses the session; incompatible with short-lived agent invocations that need to resume state. |
| **RPC over TCP** | Exposes the browser control surface to the network; Unix sockets provide file-permission-based access control and `SO_PEERCRED` peer validation (R-005), which TCP cannot. Adds unnecessary complexity for a single-host, single-user tool. |

## Consequences

**Positive:**
- Browser sessions persist across client connects/disconnects, enabling multi-turn agent workflows without re-authentication.
- Unix socket provides filesystem-based access control, restricting the control surface to the owning user.
- Direct mode remains available as a simpler path when persistence is not needed.

**Negative / Trade-offs:**
- Every public method on `CarbonylBrowser` requires a parallel implementation in `DaemonClient` and `_BrowserHandler`, creating a three-way dispatch symmetry invariant (AD-06) that must be enforced by CI.
- The daemon's `ThreadingUnixStreamServer` is not thread-safe against the underlying `CarbonylBrowser`, so concurrent clients can corrupt state (known gap, Section 10.3).
- `os.fork()` + `os.setsid()` limits the daemon to POSIX systems (Linux/macOS); Windows is unsupported.
- Daemon logs are lost after the fork stdio redirect to `/dev/null` (known gap, Section 11).
