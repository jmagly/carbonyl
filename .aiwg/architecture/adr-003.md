# ADR-003: Firefox UA Spoofing and HTTP/2 Disabling for Bot-Detection Bypass

**Status**: Accepted
**Date**: 2026-04-02
**Deciders**: roctinam

## Context

Target websites (primarily USPS.com behind Akamai Bot Manager) use a multi-signal detection chain to classify automated browsers. Akamai correlates at least three signals: (1) the JA3 TLS fingerprint, which identifies the TLS library; (2) the HTTP/2 SETTINGS frame, which has Chromium-specific window sizes and header table parameters; and (3) the User-Agent string. When all three signals indicate Chromium but the UA claims to be Firefox, the inconsistency itself becomes a detection signal. The automation layer must neutralize enough signals to pass classification without modifying the Chromium binary or its TLS stack.

## Decision

Apply two Chromium launch flags: `--user-agent=Mozilla/5.0 (X11; Linux x86_64; rv:122.0) Gecko/20100101 Firefox/122.0` to spoof the UA string in both HTTP headers and `navigator.userAgent`, and `--disable-http2` to force HTTP/1.1 and eliminate the Chromium-specific HTTP/2 SETTINGS fingerprint. Additionally, `--disable-blink-features=AutomationControlled` suppresses `navigator.webdriver=true`. Mouse movement entropy is provided via `mouse_move()` / `mouse_path()` to defeat behavioral analysis.

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|----------------|
| **Playwright stealth / puppeteer-extra-stealth** | Requires a CDP (Chrome DevTools Protocol) connection, which Carbonyl does not expose. These tools operate at the CDP layer, not the terminal/PTY layer the automation uses. |
| **TLS fingerprint randomization (curl-impersonate, mitmproxy)** | Addresses the JA3 gap but requires an external TLS-terminating proxy in the request path, adding operational complexity and a new failure domain. Deferred as a future enhancement if JA3-based blocking becomes operational. |
| **Residential proxy rotation** | Addresses IP reputation but not browser fingerprinting. Does not solve the UA/HTTP2/JA3 signal chain. Adds cost and latency. Orthogonal to the fingerprint problem. |

## Consequences

**Positive:**
- Eliminates two of three signals in the Akamai detection chain (UA string and HTTP/2 SETTINGS) using only Chromium launch flags, with no binary modifications.
- HTTP/1.1 fallback is universally supported; no site compatibility issues from disabling HTTP/2.
- `navigator.webdriver` suppression defeats the most common client-side JavaScript bot check.

**Negative / Trade-offs:**
- JA3/JA4 TLS fingerprint remains identifiably Chromium (R-002, accepted risk). Sites performing JA3 correlation can still detect the mismatch between a Firefox UA and a Chromium TLS handshake.
- HTTP/1.1 reduces connection efficiency (no multiplexing, head-of-line blocking), increasing page load times.
- The spoofed Firefox UA version (122.0) will age and eventually become implausible, requiring periodic updates.
- This is an adversarial arms race; any bypass may stop working when the target site updates its classifier.
