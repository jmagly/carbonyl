# ADR-004: Chromium User-Data-Dir for Session Persistence

**Status**: Accepted
**Date**: 2026-04-02
**Deciders**: roctinam

## Context

The automation layer must persist browser sessions (cookies, localStorage, IndexedDB, cached credentials) across process restarts so that authenticated workflows do not require re-login on every invocation. Multiple independent sessions must coexist -- for example, separate USPS accounts or different states of the same account. The persistence mechanism must integrate with Chromium's own session management rather than fighting it.

## Decision

Use Chromium's native `--user-data-dir` flag to point each named session at a dedicated profile directory under `~/.local/share/carbonyl/sessions/<name>/profile/`. The `SessionManager` provides lifecycle operations (create, fork, snapshot, restore) over these directories and maintains a `session.json` metadata sidecar. Session names are slug-validated (`^[a-z0-9][a-z0-9\-]*[a-z0-9]$`). Fork and snapshot operations use `shutil.copytree` with SingletonLock cleanup in the copy.

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|----------------|
| **Manual cookie serialization (export/import)** | Only captures cookies; misses localStorage, IndexedDB, service workers, and cached credentials. Chromium's cookie DB format is an internal SQLite schema that changes between versions, making direct manipulation fragile. Cannot reproduce full session state. |
| **In-memory-only sessions (no persistence)** | Requires re-authentication on every process restart. Incompatible with MFA workflows where re-login triggers security alerts or account lockouts (R-009). Unacceptable for the primary use case. |
| **Custom session injection via CDP** | Carbonyl does not expose a Chrome DevTools Protocol endpoint, so programmatic cookie/storage injection is not available. Even if it were, injecting partial state risks inconsistencies with Chromium's internal session tracking. |

## Consequences

**Positive:**
- Full fidelity: all Chromium session state (cookies, localStorage, IndexedDB, cache, service workers) is preserved automatically with no custom serialization.
- Fork and snapshot operations enable branching workflows -- e.g., fork a logged-in session to test different paths without re-authenticating.
- Simple implementation: the automation layer delegates all persistence to Chromium itself; `SessionManager` only manages the directory lifecycle.

**Negative / Trade-offs:**
- Chromium stores cookies and cached credentials on disk with only `--password-store=basic` protection, which is effectively plaintext (R-001, R-006). Directory permissions (`0o700`) are the primary access control.
- Chromium's `SingletonLock` prevents concurrent use of the same profile. The `clean_stale_lock()` + `open()` sequence has a narrow race condition (known gap, Section 11).
- Profile directories grow over time with cache and history data (R-010), requiring periodic cleanup or `--disk-cache-size` limits.
- `shutil.copytree` for fork/snapshot is a blocking, synchronous operation that scales linearly with profile size; large profiles with media cache can take seconds to copy.
