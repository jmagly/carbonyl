# ADR-002: pyte Virtual Terminal for Screen Parsing

**Status**: Accepted
**Date**: 2026-04-02
**Deciders**: roctinam

## Context

The automation layer must extract structured text and clickable coordinates from Carbonyl's rendered output. Carbonyl renders web pages as terminal characters over a PTY, emitting raw ANSI escape sequences. The layer needs a reliable, addressable character grid that an LLM agent can reason about using text coordinates rather than pixel positions.

## Decision

Use pyte, a Python VT100 terminal emulator library, to maintain a `Screen(220, 50)` buffer that consumes raw PTY output via `pyte.ByteStream`. All screen reads (`raw_lines()`, `extract_text()`, `find_text()`, `ScreenInspector`) operate against this in-memory buffer. The buffer is updated synchronously during explicit `drain()` calls, which poll the PTY for a caller-specified duration.

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|----------------|
| **Accessibility APIs (AT-SPI / UIAutomation)** | Carbonyl does not expose an accessibility tree. It renders directly to a terminal; there is no widget hierarchy or DOM-level accessibility interface available to the automation layer. |
| **DOM inspection via DevTools Protocol** | Carbonyl's Chromium backend does not expose a CDP (Chrome DevTools Protocol) endpoint. Adding one would require modifying the upstream Carbonyl binary, which is frozen at v0.0.3 with no active maintenance (R-008). |
| **Screenshot + OCR** | Carbonyl renders text as terminal characters, not pixels; there are no screenshots to capture. Even if a terminal screenshot were synthesized, OCR adds latency, introduces recognition errors, and cannot provide the precise character-cell coordinates needed for SGR mouse input. |

## Consequences

**Positive:**
- Provides a reliable, addressable character grid with exact column/row positions that map directly to SGR mouse coordinates for click targeting.
- Handles all ANSI escape sequences (cursor movement, colors, alternate screen buffer) correctly, without custom parsing.
- Zero external service dependencies -- pyte is a pure-Python library with no native extensions.
- Works with any content Carbonyl can render, since it operates on the terminal output layer rather than the page DOM.

**Negative / Trade-offs:**
- The screen buffer is only updated during explicit `drain()` calls; there is no event-driven notification when a page finishes loading (known gap, Section 11).
- Unicode block characters (U+2580-U+259F) used by Carbonyl for sub-cell rendering obscure the underlying text, requiring `extract_text()` to filter them out and losing visual layout information.
- pyte is single-threaded, making the screen buffer update path synchronous and incompatible with concurrent reads during a drain.
- The automation layer can only see what Carbonyl renders to the terminal grid; off-screen content, hidden elements, and non-visual DOM state are invisible.
